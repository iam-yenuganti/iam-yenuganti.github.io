AI Agents in Your IDE: From First Install to Agentic Build & Deploy Workflows

⚡ What you will be able to do after reading this

By the end of this guide you will have:

  • GitHub Copilot and Claude both wired into VS Code and talking to your codebase
  • MCP (Model Context Protocol) servers connected so your AI agent can read files, run terminals, query Azure, and call APIs
  • A working agentic workflow that writes Terraform → validates → plans → commits → triggers a pipeline — all inside the IDE
  • Concrete prompts for each stage: scaffold, review, fix, document, deploy
  • A clear understanding of what each tool does well and where the boundaries are

There is a difference between having an AI assistant installed and actually using it as an agent. Most developers install GitHub Copilot, accept inline completions for a week, and conclude it saves maybe 20 minutes a day. Architects who wire up the full stack — multiple models, MCP tool servers, agent mode, workspace context — are running multi-step autonomous workflows inside the same IDE window. This guide shows you exactly how to get there.

Section 1 — The Full Tool Map

What every AI tool actually does inside your IDE

Before wiring anything up, it helps to know what role each tool plays. These are not interchangeable — they solve different problems and complement each other.

GH
GitHub Copilot Enterprise
Native VS Code & JetBrains integration. Inline completions, Copilot Chat panel, agent mode, workspace-wide context (@workspace), and slash commands (/fix, /explain, /tests). The workhorse for day-to-day coding assistance.
Best for: inline completions, multi-file edits, PR summaries, test generation
CL
Claude (via Copilot or Cursor) Paid
Anthropic’s Claude Sonnet/Opus models, accessible through GitHub Copilot’s model picker or Cursor’s native integration. Excels at long-context reasoning — paste an entire Terraform module or architecture document and ask deep analytical questions.
Best for: architecture reviews, ADRs, complex multi-file reasoning, long docs
CR
Cursor Paid
A VS Code fork with AI baked into the editor at a deeper level than any plugin. Model selector (GPT-4o, Claude, Gemini), @codebase semantic search, Composer for multi-file edits, and native MCP support. Best for greenfield work where you want maximum AI surface area.
Best for: new projects, multi-file refactors, prototype-speed delivery
GM
Gemini Code Assist Paid
Google’s VS Code extension. Strong on Google Cloud Platform resources (GKE, BigQuery, Cloud Run). Useful if your stack spans multi-cloud and you need GCP-specific Terraform or deployment YAML generated accurately.
Best for: GCP-specific IaC, multi-cloud teams
CF
Amazon Q Developer Free Tier
AWS’s IDE extension for VS Code and JetBrains. Free tier for individuals, paid Pro tier adds security scanning and multi-repo context. Best AWS CloudFormation and CDK generation quality of any tool tested.
Best for: AWS CDK, CloudFormation, Lambda-heavy stacks
MC
MCP Tool Servers Open Source
Model Context Protocol servers that give your AI agent real tools: read/write files, execute terminal commands, call REST APIs, query databases, browse the web, call Azure CLI. The bridge between “AI suggests code” and “AI actually does the work.”
Best for: agentic workflows, build & deploy automation, tool orchestration
💻 How I use these together in VS Code

My daily setup: GitHub Copilot for inline completions and quick fixes; Claude (via Copilot model picker) for architecture discussions and long-context reviews; MCP servers for the Azure CLI, filesystem, and Git tools so Copilot agent mode can actually run commands, not just suggest them. Cursor gets used for new module greenfield builds where I want Composer’s multi-file edit capability.

Section 2 — Install & Wire Up: VS Code

Complete setup from scratch — Copilot, Claude, and MCP in one window

2.1 — GitHub Copilot

Install the extension
Open VS Code Extensions panel (Ctrl/Cmd + Shift + X), search GitHub Copilot, install both GitHub Copilot and GitHub Copilot Chat. Sign in with your GitHub account when prompted.
Verify inline completions are active
Open any file, start typing — grey ghost text should appear. Tab to accept, Alt+] to cycle alternatives. If nothing appears: Ctrl/Cmd + Shift + P → GitHub Copilot: Enable.
Open Copilot Chat panel
Click the chat bubble icon in the Activity Bar (left sidebar) or press Ctrl/Cmd + Alt + I. This is where you interact with the agent — ask questions, run slash commands, trigger multi-file edits.
Switch to Agent Mode
In the Copilot Chat input, click the dropdown next to the send button and select Agent. In agent mode, Copilot can iterate: run commands, read error output, fix the code, re-run. It does multi-step work without you prompting each step.
Select Claude as your model
In the Copilot Chat panel, click the model name at the bottom of the input box. Select Claude Sonnet 4.5 or Claude Opus (available on Copilot Pro/Enterprise). The model switch is per-conversation — you can use GPT-4o for quick edits and Claude for architecture discussions in the same session.
💡 Copilot Chat Context Commands
  • @workspace — gives the model context of your entire open repo, not just the current file
  • @vscode — asks about VS Code settings, keybindings, extensions
  • #file:path/to/file.tf — explicitly attaches a specific file to the context
  • /fix, /explain, /tests, /doc — slash commands for targeted tasks

2.2 — Connect MCP Servers (the agentic power-up)

MCP (Model Context Protocol) is the open standard that lets AI agents call external tools. Without it, Copilot can only suggest commands. With MCP, it can run them. Here is how to wire up the most useful MCP servers for cloud architecture work:

📄 .vscode/mcp.json — add to your project root 
{
  "servers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem",
               "/path/to/your/project"]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_PERSONAL_ACCESS_TOKEN": "${env:GITHUB_TOKEN}" }
    },
    "azure-cli": {
      "command": "npx",
      "args": ["-y", "mcp-server-azure-cli"]
    },
    "terminal": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-shell"],
      "env": { "SHELL_ALLOW_LIST": "terraform,az,git,kubectl" }
    }
  }
}

Once this file is in place and VS Code reloads, Copilot Agent mode gains four new tools: read/write files, call GitHub APIs, run Azure CLI commands, and execute approved terminal commands. The agent can now do things like: check the current Azure subscription, list existing resource groups, scaffold Terraform, validate it, and commit — all from a single chat prompt.

⚠ Security: Shell MCP server

Always use SHELL_ALLOW_LIST to restrict which commands the MCP shell server can execute. Never configure it with unrestricted shell access. The agent will only call commands on the allow list — this is your blast-radius control.

2.3 — Add the workspace instructions file

This is the single highest-leverage setup step for IaC and cloud work. Create .github/copilot-instructions.md at the repo root. Every Copilot prompt in that workspace automatically includes this context — your naming conventions, standards, and constraints become the model’s defaults without repeating them in every prompt.

📄 .github/copilot-instructions.md — template for IaC repos 
## Project
Azure landing zone Terraform repo for [Organisation].

## Naming Convention
Pattern: {env}-{region-short}-{service}-{instance}
Example: prod-eus2-kv-01
Regions: eastus2=eus2, westus2=wus2, centralus=cus

## Technology Stack
- IaC: Terraform, azurerm provider >= 3.90.0
- CI/CD: GitHub Actions with OIDC (no client secrets)
- Secrets: Azure Key Vault (never hardcode)
- Container platform: AKS 1.30+

## Non-negotiable Defaults
- All resources tagged: Environment, Owner, CostCenter, Project
- Key Vault: purge_protection=true, soft_delete_retention=90, RBAC model
- Storage: minimumTlsVersion=TLS1_2, allowBlobPublicAccess=false
- Networking: deny-by-default NSGs, no 0.0.0.0/0 inbound rules

## Repo Layout
modules/  environments/prod/  environments/nonprod/  .github/workflows/

Section 3 — JetBrains Setup (IntelliJ, GoLand, PyCharm)

Same models, different IDE — what changes and what doesn’t

Install GitHub Copilot plugin
File → Settings → Plugins → Marketplace → search GitHub Copilot. Install and restart. The same Copilot Chat panel, agent mode, and model picker work identically to VS Code.
MCP in JetBrains
JetBrains AI Assistant (separate product) supports MCP server configuration via Settings → Tools → AI Assistant → Model Context Protocol. GitHub Copilot’s JetBrains plugin is adding MCP support in 2026 — check the plugin settings for an MCP tab after the latest update.
copilot-instructions.md works the same
The .github/copilot-instructions.md file is read by the Copilot plugin regardless of IDE. Your workspace conventions are carried over automatically — no duplicate configuration needed.
For Go / Python / Java workloads
JetBrains IDEs provide the richest language server support for these languages. Copilot completions in IntelliJ/GoLand will be significantly more accurate for Go microservices or Python ML code than VS Code because the IDE’s type inference provides better context to the model.

Section 4 — Real Scenarios: What I Actually Ask the Agent

Exact prompts and what the agent does — from IaC to deploy

These are real prompt patterns I use in VS Code Copilot Agent mode for cloud architecture work. Each shows the prompt, the tools the agent invokes, and the outcome.

👨‍💻 You (Copilot Agent mode, @workspace active)
Create a Terraform module for an Azure Storage Account. Use my naming convention, azurerm >= 3.90.0. - Standard_LRS, StorageV2, TLS 1.2, no public blob access - Network: deny by default, allow AzureServices - System-assigned managed identity - Diagnostic settings to the Log Analytics workspace in my variables Validate it, fix any errors, then show me what terraform plan would do.
🤖 Agent (tools used: filesystem write, terminal: terraform validate)
Creating modules/storage-account/main.tf ... Creating modules/storage-account/variables.tf ... Creating modules/storage-account/outputs.tf ... Running: terraform validate [output: Success! The configuration is valid.] Terraform plan summary: + azurerm_storage_account.this (create) + azurerm_monitor_diagnostic_setting.storage (create) No resources will be destroyed.
✅ What happened
The agent wrote three files, ran terraform validate autonomously, read the output, confirmed success, then described what a plan would show — all in one prompt. No copy-pasting, no terminal-switching.
👨‍💻 You
@workspace Review all Terraform files in environments/prod/ for CIS Azure Benchmark v2 violations. List each finding as: file, line, control ID, risk, fix.
🤖 Agent (tools used: filesystem read, @workspace index)
Scanned 14 files. Found 3 issues: 1. environments/prod/storage.tf:23 Control: DP-3 (Encryption in Transit) Risk: supportsHttpsTrafficOnly not explicitly set (defaults to true but should be explicit for audit evidence) Fix: add supportsHttpsTrafficOnly = true 2. environments/prod/network.tf:41 Control: NS-1 (Network Segmentation) Risk: NSG rule allows inbound 3389 from 0.0.0.0/0 (RDP open to internet) Fix: restrict source to your corporate IP range or remove entirely 3. environments/prod/aks.tf:88 Control: PA-7 (Privileged Access) Risk: local_account_disabled not set to true Fix: add local_account_disabled = true to azurerm_kubernetes_cluster
✅ What happened
One prompt replaced a manual security review that typically takes 30–60 minutes. The agent read every file in the directory, cross-referenced against a known benchmark, and returned structured findings you can copy directly into a security review report.
👨‍💻 You (with GitHub MCP server connected)
Write a GitHub Actions workflow that: 1. Runs on PR to main 2. Authenticates to Azure using OIDC (no client secrets) 3. Runs terraform fmt, validate, and plan 4. Posts the plan output as a PR comment 5. On merge to main, runs terraform apply Use my subscription ID from the repo secrets. After creating the file, create a feature branch and open a draft PR.
🤖 Agent (tools used: filesystem write, GitHub MCP: create branch, create PR)

✅ What happened
The agent wrote the pipeline YAML, created a Git branch, committed, and opened a draft PR in GitHub — all from one prompt, without leaving VS Code. The GitHub MCP server handled the API calls. You review the PR in your browser, the code is already there.

Section 6 — The End-to-End Agentic Build & Deploy Workflow

One prompt chain that takes you from idea to deployed infrastructure

Here is the full workflow I walk through for a new Azure service deployment, using Copilot Agent mode with filesystem, shell, and GitHub MCP servers connected:

📄
Scaffold
Terraform module generated from prompt
›
✅
Validate
Agent runs terraform validate, fixes errors
›
🔍
Security Review
CIS scan prompt, agent reads all files
›
📄
Document
README + variable docs generated
›
🚀
PR
Agent branches, commits, opens draft PR
›
🛠️
Pipeline
Plan runs on PR, apply on merge

Each step is a separate prompt, but they are issued in sequence in the same Copilot Chat conversation. The agent maintains context across the conversation — it knows what it created in step 1 when it is reviewing in step 3.

Step 1: Scaffold
Create a Terraform module for Azure Container Registry. Use my naming convention, standard tier, geo- replication to westus2, admin user disabled, CMK encryption using the Key Vault in my variables. Validate after creating.
What agent does
Writes main.tf, variables.tf, outputs.tf to modules/container-registry/ Runs: terraform validate Output: Success Reports: 3 resources will be created (azurerm_container_registry, azurerm_container_registry_replication, azurerm_container_registry_encryption)
Agent wrote + validated in ~45 seconds. You get working Terraform with your actual naming convention, not a generic skeleton.
Step 3: Security Review
Review the module we just created against CIS Azure Benchmark v2. List any non-compliant settings with control IDs and fixes.
What agent does
Reads modules/container-registry/main.tf Finding: NS-2 — quarantine_policy_enabled not set to true (recommended for untrusted image isolation) Fix: add quarantine_policy_enabled = true No other issues found. 2 resources fully CIS-compliant.
One finding — caught before the PR is even opened. Control ID is ready to paste into your security review documentation.
Step 4: Document
Generate a README.md for this module. Include: purpose, inputs table, outputs table, usage example, security notes, and compliance controls this module satisfies.
What agent does
Reads all .tf files in the module. Writes modules/container-registry/README.md Includes: - Auto-generated inputs/outputs tables - Terraform usage example block - Security notes: CMK, no admin user, geo-replication - Compliance: CIS 7.4, 7.5, MS-1
Documentation that would normally be skipped entirely gets generated in seconds from the code that already exists.

Section 7 — Dos, Don’ts & Common Mistakes

What separates architects who trust AI from those who get burned by it

✅ Do These
  • Always use agent mode for multi-step tasks — regular chat suggests, agent mode executes, validates, and iterates
  • Keep the copilot-instructions.md up to date — treat it like a configuration file, not a one-time setup. Review it when standards change.
  • Switch models for different tasks — GPT-4o for quick completions, Claude for long architecture discussions. Use the model picker actively.
  • Let the agent run terraform validate first — before reviewing logic, catch schema errors automatically
  • Use @workspace for review tasks — reviewing a single file misses cross-file dependencies; workspace context catches them
  • Review IAM and network rules by hand even when the agent generates them — these are the highest blast-radius outputs
  • Use the GitHub MCP server to open PRs from the IDE — eliminates the context-switch to the browser mid-flow
❌ Don’t Do These
  • Never paste secrets, connection strings, or API keys into the chat window — replace with [REDACTED] or variable names
  • Don’t configure the shell MCP server without an allow list — unrestricted shell access to an AI agent is a serious security risk
  • Don’t run terraform apply from agent mode on production — plan is fine; apply requires human review of the plan output first
  • Don’t skip terraform validate — the model invents property names that look right but don’t exist. Validate catches them in seconds.
  • Don’t use vague prompts on complex tasks — “create a Kubernetes deployment” generates a demo. “Create an AKS Deployment for my payment-service container at image:tag, 3 replicas, resource limits 500m/512Mi, liveness probe on /health:8080” generates production-ready YAML.
  • Don’t assume the agent has current provider knowledge — always specify the azurerm/provider version in your prompt and run validate
🔧 Real-World Mistake I Made

I asked Copilot Agent to “connect the Key Vault module to the storage account module.” It wired up the dependency correctly but generated an azurerm_key_vault_key resource with a rotation_policy block using a property that was deprecated two provider versions ago. The code looked right, passed my eye-scan, and failed terraform validate immediately. The lesson: always run validate before any visual review, because schema errors are invisible to the eye.
